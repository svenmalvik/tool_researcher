# Tool Research Intake

## Overview

Use this skill to scope the research before the project moves into market analysis. The output of this skill is a research brief that is specific enough to guide the baseline and analysis work without dragging the intake out.

## Workflow

1. Confirm the candidate tool and the decision the user is trying to make.
2. Do a lightweight orientation pass on the candidate tool and category before asking the main intake questions.
3. Use that orientation pass to select the next intake questions from `references/question-bank.md` and keep the planned intake sequence short.
4. Ask one broad question per turn. Keep the main intake to a short bulk of 3 to 4 questions when possible, and do not summarize, reason, or recommend until that bulk is complete.
5. Explicitly ask what similar tools are already in use today.
6. Capture what is working well and poorly with the current tools.
7. Ask who the primary users are and what success looks like.
8. Ask whether the user has any additional information, documents, links, or prior analysis that should shape the research.
9. Create an initial task list for the full workflow.
10. Turn the answers into a concise research brief with clear open questions.

## Intake Style

- Start with a narrow orientation pass: enough to classify the tool and tailor the intake, not enough to become full research.
- Prefer one question per turn, but make each question broad enough that the user can answer in a useful paragraph.
- Keep the main intake short, usually 3 to 4 turns before any summary.
- Cover the essentials first: scope, decision, users, baseline, constraints, and existing material.
- Keep the first turn short enough that the user can answer without scrolling fatigue.
- Do not summarize, interpret, or give provisional recommendations between intake answers.
- Do not ask about pricing, budget, or expected seat count during intake. If the user volunteers commercial context, capture it without reopening the intake sequence.
- Use follow-up questions only to close important gaps such as approval owner, success criteria, migration barriers, or category-specific risks.
- Summarize what you learned only after the planned intake bulk is complete.

## Required Output

Produce a research brief with these sections:

- Candidate tool
- Problem to solve
- Why now
- Decision and timeline
- Users and stakeholders
- User roles
- Success criteria
- Current tools in use
- Pain points and desired outcomes
- Constraints and non-negotiables
- Procurement or migration constraints already known
- Decision criteria
- User-provided context and source material
- Open questions
- Initial task list

## Handoff

If current tools are already in use, hand the brief to `tool-research-baseline` next.

If there is no current tool baseline, state that the evaluation is effectively greenfield and hand the brief to `tool-research-analysis`.
