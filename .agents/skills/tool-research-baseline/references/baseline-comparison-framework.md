# Baseline Comparison Framework

Research the current tools first and summarize them with the dimensions below.

## Required Dimensions

- Primary use cases covered today
- Teams, users, or workflows that depend on the tool
- Strengths the organization would lose if the tool were removed
- Pain points, friction, or unmet needs
- Important integrations and dependencies
- Number and type of users on the current tool
- Pricing or licensing facts if they affect switching
- Switching costs and migration complexity
- Risks of keeping the current tool

## Output Expectations

The baseline should answer these questions:

- What is the organization getting today?
- What is not working well enough?
- What is the current spend or licensing shape for the current tools?
- What would a new tool need to improve materially?
- What would be expensive or risky to replace?

## Suggested Table Columns

Use a table with columns such as:

- Tool
- Current role
- Strengths
- Weaknesses
- Users and roles
- Pricing model
- Integrations
- Switching cost
- Keep or replace pressure

## Greenfield Case

If no meaningful incumbent tools are in use, record that as a greenfield baseline instead of forcing a weak comparison.
